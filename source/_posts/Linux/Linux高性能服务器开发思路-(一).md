笔者最近在阅读 Linux 服务器开发时，阅读了游双先生编著的《Linux高性能服务器编程》这本书后，感觉整本书的信息量很大，读完后能体会到这本书的思路由浅入深地很清晰，但是笔者感觉还是有些地方没有悟透，因此写下这篇文章来梳理一下一个高性能的服务器究竟“高性能”在哪些地方，又是通过什么样的思路和方法来进行优化的。

### 服务器组成概览

首先来看看服务器本身的组成，服务器可以解构成三种主要模块，先概览一下：

- **I/O处理单元**：主要管理客户连接，数据收发不一定在此进行（主要取决事件处理方式）
- **逻辑单元**：分析并处理客户数据
- **存储单元（可选）**：可以是数据库缓存和文件

从模块设计的角度来说，这三种模块承担的职责在单个服务器程序上和服务器集群上都是一样的，只是载体不同，对于单个而言就是一个程序的模块，对于集群而言则是一个专门负责这个功能的服务器或服务器组。

对于一个服务器或服务器组，单元与单元之间的通信主要靠 **请求队列**，请求队列是各单元之间的通信方式的抽象。

请求队列通常被实现为池的一部分，池是一组资源的集合，在服务器启动时候就创建好并初始化，说白了就是空间换时间，用服务器的硬件资源换取效率，用大量的静态资源来避免动态分配造成的开销。

### I/O处理单元优化

首先来看 I/O 处理单元是如何提升性能的，为了避免非阻塞 I/O 造成性能的极大浪费，即事件没准备好就别搁那一直问“好了吗”，等到事件发生了再进行操作，解决这个问题就需要非阻塞 I/O 搭配 **I/O复用** 和 **SIGIO信号** 这两种通知机制来解决这个问题。

**I/O复用** 是最常使用的 I/O 通知机制。它指的是，应用程序通过 I/O 复用函数向内核注册一组事件，内核通过 I/O 复用函数把其中就绪的事件通知给应用程序。Linux 上常用的 I/O 复用函数是 `select`、`poll` 和 `epoll_wait`，需要指出的是，I/O 复用函数本身是阻塞的，它们能提高程序效率的原因在于它们具有同时监听多个 I/O 事件的能力。

**SIGIO信号** 也可以用来报告 I/O 事件。6.8 节的最后一段提到，我们可以为一个目标文件描述符指定宿主进程，那么被指定的宿主进程将捕获到 SIGIO 信号。这样，当目标文件描述符上有事件发生时，SIGIO 信号的信号处理函数将被触发，我们也就可以在该信号处理函数中对目标文件描述符执行非阻塞 I/O 操作了。关于信号的使用，我们将在第 10 章讨论。

上述的处理事件方式统属 **同步 I/O 模型**，即 I/O 就绪事件发生，通知程序应该执行。

而 **异步 I/O 模型** 则是 I/O 就绪事件发生，内核执行，立刻返回，通知程序执行完成。

### 事件处理方式

这里涉及到 I/O 同步异步的问题，以此衍生出了两种不同的事件处理方式：

#### Reactor模式

从名字可以看出一点这个模式的思想，“响应者”（笔者瞎翻译的，仅用于理解），主线程只负责响应变化。

即主线程只负责监听文件描述是否有事件，有事件就通知工作线程，除此之外主线程不做任何实质性的工作，I/O 操作在工作线程中进行，工作线程做什么取决于事件的类型，所以没有必要区分读/写工作线程。

#### Proactor模式

同样该名字也可以看出一点这个模式的思想，“主动者”（笔者瞎翻译的，仅用于理解），主线程和内核主动承担所有的 I/O 操作。

即工作线程只负责处理业务逻辑，读写都依赖接收来自于内核的信号，根据工作线程中定义的信号函数进行处理。

### 服务器并发模式

刚才说的是 I/O 处理单元的优化方式，让我们回到服务器的层面，看一下服务器并发模式。

并发模式是指 I/O 处理单元和多个逻辑单元之间协调完成任务的方法，主要有以下两种模式：

- **半同步/半异步模式**：主要指的是对于逻辑单元来讲使用同步线程处理客户逻辑，对于 I/O 处理来讲需要异步线程用于处理 I/O 事件。
- **领导者/追随者模式**：程序仅有一个领导者线程，负责监听 I/O 事件，其他线程都是追随者，休眠在线程池中等待成为新的领导者。当前的领导者如果检测到 I/O 事件通知线程池选取新的领导者，然后自己处理 I/O 事件，新的领导者进行监听事件，笔者理解就是管事没事时候看看有没有活要干，有活了就找个人来当管事的，自己忙去了，如此循环下去。

### 逻辑单元优化

接下来看看逻辑单元是如何提升性能的，主要就是靠有限状态机。

#### 有限状态机（FSM）的核心价值

网络协议（如 HTTP、SMTP、Redis 或自定义二进制协议）本质上是状态驱动的。逻辑单元需要将连续的字节流安全、高效地解析为结构化数据，FSM 正是解决该问题的标准范式：

- **消除深层嵌套与隐式状态**：传统 `if-else` 解析逻辑随协议复杂度指数级膨胀，FSM 将解析过程抽象为 `当前状态 + 输入字符 → 下一状态 + 执行动作` 的确定性映射，代码可维护性与可测试性显著提升。
- **减少数据拷贝**：FSM 可直接在 I/O 接收缓冲区上逐字节推进状态，配合环形缓冲或 `readv/writev`，避免反复 `malloc/memcpy`，降低内存分配压力。
- **快速失败与资源回收**：每个状态可独立设置超时/非法字符阈值，一旦检测到畸形包或状态跳变，立即触发错误分支并关闭连接，防止恶意请求长期占用逻辑线程。

#### 逻辑单元的其他关键优化方向

除 FSM 规范协议解析路径外，逻辑单元的性能瓶颈通常集中在任务调度、内存访问与并发控制上，常见工程优化手段包括：

1. **无锁/分片任务队列**：逻辑单元从请求队列拉取任务时，优先采用基于 CAS 的环形队列或按 CPU 核心分片的局部队列，避免全局互斥锁引发的线程竞争与上下文切换。
2. **CPU 缓存友好设计（Cache Locality）**：将连接上下文、会话状态、统计计数器等热点数据紧凑布局（如使用结构体数组而非指针链表），减少 Cache Miss；避免跨线程传递大对象或频繁触发伪共享（False Sharing）。
3. **批处理与合并执行**：高并发场景下，将多个小请求合并为一次批量操作（如批量写 Binlog、批量刷新 Redis Pipeline、合并系统调用），可大幅摊薄单次处理的固定开销。
4. **工作窃取调度（Work-Stealing）**：当线程池负载不均时，空闲线程可主动从繁忙线程的任务队列尾部“窃取”任务执行，消除长尾延迟，提升整体 CPU 利用率。
